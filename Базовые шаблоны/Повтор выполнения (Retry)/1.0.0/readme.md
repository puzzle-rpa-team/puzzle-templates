# Retry (повтор выполнения)

Шаблон для выполнения действий с повторными попытками при возникновении ошибок.

Позволяет повысить устойчивость процессов и стандартизировать обработку нестабильных операций.

---

## Назначение

Используйте данный шаблон, если вам нужно:

- повторять выполнение действия при ошибке  
- повысить надёжность работы с внешними системами  
- логировать ошибки каждой попытки  
- контролировать количество попыток выполнения  
- получать детализированную информацию о сбоях  

---

## Входные параметры

Функция принимает параметры, управляющие логикой повторов:

| Параметр      | Тип        | Обязательный | Описание |
|--------------|-----------|--------------|----------|
| `params`      | any       | нет          | Входные данные |
| `max_retries` | int       | да           | Максимальное количество попыток |
| `retry_delay` | int/float | да           | Задержка между попытками (сек) |

- `params` — входные данные для выполнения основной логики
- `max_retries` — максимальное количество попыток выполнения  
- `retry_delay` — задержка между попытками (в секундах)  

Рекомендуется:

- задавать разумное количество попыток (обычно 2–5)  
- использовать задержку при работе с API и UI  
- не использовать слишком большие значения `max_retries` без необходимости  

---

## Структура шаблона

Шаблон состоит из 5 основных частей:

### 1. Инициализация результата

В переменной `execution_result` сохраняется результат выполнения.

Структура включает:

- `status` — статус выполнения  
- `message` — текстовое описание результата  
- `data` — данные выполнения  
- `error_description` — описание последней ошибки  

---

### 2. Цикл повторных попыток

Выполнение основной логики происходит внутри цикла:

- выполняется до `max_retries` раз  
- каждая попытка оборачивается в обработку ошибки  

---

### 3. Основная логика

В данном блоке размещается действие, которое необходимо повторять.

Примеры:

- HTTP-запрос  
- работа с UI  
- чтение/запись файлов  
- вызов внешних систем  

Рекомендуется:

- размещать только одну логическую операцию  
- избегать лишней вложенности  
- не смешивать retry и бизнес-логику  

---

### 4. Обработка ошибки

При возникновении ошибки:

- увеличивается счётчик попыток  
- формируется сообщение об ошибке  
- ошибка добавляется в список `data.errors`  
- ошибка записывается в лог  
- выполняется задержка перед следующей попыткой (если есть попытки)  

---

### 5. Завершение выполнения

#### При успехе:

- выполнение прерывается  
- возвращается результат  

#### При исчерпании попыток:

- устанавливается статус `error`  
- формируется итоговое сообщение  
- возвращается информация о всех ошибках  

---

## Возвращаемое значение (`execution_result`)

Функция возвращает словарь `execution_result`.

### Рекомендуемая структура:

```json
{
  "status": "success | error",
  "message": "string",
  "data": {
    "attempts_used": 0,
    "result": "",
    "errors": []
  },
  "error_description": "string"
}
```

### Поля:

- `status` — `success` / `error`  
- `message` — описание результата  
- `data`:
  - `attempts_used` — количество попыток  
  - `result` — результат выполнения  
  - `errors` — список ошибок  
- `error_description` — последняя ошибка  

---

## Когда использовать

- нестабильные API  
- UI-автоматизация  
- работа с файлами и сетью  

---

## Когда не использовать

- простые вычисления  
- валидация данных  
- детерминированная логика  

---

## Особенности

- шаблон не содержит бизнес-логики  
- работает как обёртка над действиями  

---

## Примеры использования

- повтор HTTP-запроса  
- повтор клика по UI  
- повтор чтения файла  
- повтор обращения к API
