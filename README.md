# Puzzle RPA Templates

Репозиторий содержит набор шаблонов для разработки роботов в **Puzzle RPA**.

Шаблоны предназначены для ускорения разработки, стандартизации логики и обеспечения единообразного подхода при создании автоматизаций.

---

## 📌 Назначение

Репозиторий используется для:

* хранения и распространения шаблонов
* повторного использования готовых решений
* стандартизации разработки роботов
* упрощения внедрения и поддержки решений

---

## 🧩 Что такое шаблон

Шаблон — это готовая структура функции или группы блоков, которая:

- реализует типовой сценарий (например: retry, HTTP-запрос, работа с Excel)  
- может использоваться как самостоятельная функция или как фрагмент процесса  
- содержит документацию и метаданные  
- может быть встроена в Studio и переиспользована

---

## 📂 Структура репозитория

```text
templates/
 ├── categories.json
 ├── <category>/                   # Категория шаблона
 │    ├── templates.json
 │    └── <template>/              # Название шаблона
 │         ├── versions.json       # Версии шаблона
 │         └── <version>/
 │              ├── template.json  # Описание структуры блоков
 │              ├── readme.md      # Документация шаблона
 │              ├── preview.png    # Превью в Studio
 │              ├── meta.json      # Метаданные
 │              └── resources/
 │                   └── masterUI/
 │                        ├── elements/
 │                        │    └── <id_name>.json
 │                        ├── meta/
 │                        │    └── targetProperties.json
 │                        └── screens/
 │                             └── <id>.png
```

## ⚙️ Общие правила разработки шаблонов

### 1. Тип шаблона

Шаблон может быть одного из двух типов:

- **функция** — самостоятельная логика с параметрами и возвращаемым значением  
- **группа блоков** — переиспользуемый фрагмент процесса без обязательного возвращаемого значения  

---

### 2. Возвращаемое значение для шаблонов-функций

Если шаблон реализован в виде функции, рекомендуется использовать единый формат результата `execution_result`:

```json
{
  "status": "success | error",
  "message": "string",
  "data": {},
  "error_description": "string"
}
```
---

Это позволяет:

- стандартизировать обработку результата
- упростить переиспользование функций
- обеспечить единый подход к ошибкам и логированию

### 3. Обработка ошибок

Для шаблонов-функций рекомендуется:

- не завершать выполнение необработанным исключением
- возвращать информацию об ошибке в execution_result
- логировать ошибки

Для шаблонов-групп блоков подход к обработке ошибок определяется сценарием использования.
---

### 4. Поддержка повторных попыток (при необходимости)

Для нестабильных операций (API, UI, сеть):

* использовать retry-подход
* задавать `max_retries` и `retry_delay`

---

### 5. Читаемость и простота

* использовать понятные имена переменных
* избегать избыточной логики
* минимизировать побочные эффекты

---

### 6. Документация обязательна

Каждый шаблон должен содержать:

* описание назначения
* входные параметры
* структуру
* рекомендации

---

### 7. Совместимость с Studio

* шаблон должен корректно генерироваться в Python
* не использовать нестабильные или недоступные зависимости
* учитывать ограничения корпоративных сред

---

### 8. Расширяемость

* структура шаблона должна позволять доработки
* не “зашивать” жёсткую бизнес-логику
* предусматривать переиспользование

---

## 📦 Версионирование

Каждый шаблон поддерживает версии:

* изменения фиксируются в `versions.json`
* новая версия создаётся в отдельной папке
* старые версии сохраняются для обратной совместимости

---

## 🧠 Рекомендации

* сначала проектируйте контракт результата
* затем структуру логики
* только потом реализацию
* старайтесь делать шаблоны максимально универсальными

### Выбор типа шаблона

Рекомендуется использовать:

- **функцию**, если у логики есть чёткие входные параметры и возвращаемый результат  
- **группу блоков**, если шаблон представляет собой фрагмент сценария, встраиваемый в процесс
